# JDK vs JRE vs JVM
JVM 
- Stands for Java Virtual Machine. 
- The engine that runs Java bytecode.
- Analogy: Java Program = car, JVM = Engine
- Without JVM, Java cannot run.

## Java bytecode?
Java Bytecode is the intermediate code generated by the Java compiler (javac) from your .java source file.
Flow:
Employee.java
|
javac
|
Employee.class   ← Bytecode
|
JVM
|
Machine Code
|
Execution

## JRE?
- JRE stands for Java Runtime Environment.
- JRE contains : JVM and libraries needed to run Java application.
- Example: A ready-to-drive car.
- You can run programs but cannot build them.

## JDK?
- JDK stands for Java Development Kit.
- Contains: JRE, Complier(Java), Debugging tools, and Development utilities.
- Complete car factory. you can create and run applications.

JVM executes java bytecode. JRE provides runtime environment containing JVM and libraries.
JDK provides development tools along with JRE for building Java applications.

OS / CPU
Think of JVM as a company office:
- class loader - HR (brings employees into company)
- memory area - Workspaces
- Execution Engine - Employees doing work
- Garbage Collector - Housekeeping team
Big picture:
- Suppose the code runs. What happens? The JVM starts.
- Class loader loads classes into memory
- Memory is allocated. JVM creates memory areas.(Heap, Stack,Metaspace, PC Register, Native Stack)
- Object is created. Reference variable is stored in stack. Object is stored in heap.
- Execution engine reads bytecode `new Employee()` and executes it.
- Garbage Collector cleans memory. Object becomes unreachable. GC removes it.
